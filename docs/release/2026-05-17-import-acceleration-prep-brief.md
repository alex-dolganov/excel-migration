# Import Acceleration Prep Brief

## Цель текущего этапа

Сейчас задача не в полном прод-роллауте, а в безопасной подготовке импорта к большим объемам при локальной проверке на одном портале. Внедряем только те изменения, которые:

- ускоряют импорт уже сейчас
- не меняют бизнес-логику импорта
- не ломают текущие сценарии simple/advanced import
- не требуют сразу публикации через облачных администраторов

Остальное на этом этапе доводим до состояния готовности: конфиги, документация, release artifacts, наблюдаемость и handoff-пакет.

## Что делаем сейчас

### 1. Безопасное ускорение текущего импорта

- убрать безусловную задержку на каждую строку в Python import execution
- оставить adaptive retry/backoff только на реальные rate-limit ошибки Bitrix24
- уменьшить частоту сохранения прогресса в БД
- сохранить текущую логику:
  - mapping
  - dedup
  - duplicate decisions
  - linked import
  - import reports

**Ожидаемый эффект:** заметное ускорение без изменения пользовательского сценария и без тяжелого рефакторинга.

### 2. Локальная проверка на одном портале

- проверить preview
- проверить validation
- проверить run
- проверить queue-enabled background run
- проверить imports для целевых сущностей на большом CSV
- отдельно проверить, что simple import не ломает advanced import

**Ожидаемый эффект:** подтверждаем, что ускорение не сломало существующий импорт.

### 3. Минимальная observability-подготовка

- добавить trace/job correlation в import session summary
- подготовить telemetry точки для preview/validate/run
- не делать пока глубокую интеграцию support automation, если это задерживает безопасный rollout

**Ожидаемый эффект:** после локальной проверки уже видно, где тормозит импорт, и это можно будет использовать в следующем этапе.

## Что пока только готовим

### 4. Подготовка production-конфигурации

- `.env.example` для внешних MySQL/RabbitMQ/OTLP
- `docker-compose.yml` как локальный reference
- SQL/init notes для MySQL
- release checklist
- Harbor release template
- admin handoff template

### 5. Подготовка к безопасной публикации

- описание проекта
- шаблон задачи на Максима (`horokey`)
- шаблон задачи для облачных администраторов
- правила по домену, тегу образа и приложенным конфигам
- требования по Harbor scan: без Critical/High

### 6. Подготовка следующего технического этапа

Пока не внедряем в основной поток, но проектируем:

- parse-once staging
- chunk processing
- идемпотентные chunk retries
- экономный XLSX parsing
- support ticket proxy через внутреннюю прослойку

Это следующий этап после подтверждения, что fast safe optimization не ломает текущий импорт.

## Что сейчас не трогаем без отдельного решения

- полную смену архитектуры preview/validate/run
- радикальный рефакторинг dedup flow
- массовую перестройку linked import
- продовую публикацию образа
- rollout через облачных администраторов

## Критерий готовности текущего этапа

- импорт работает так же, как сейчас, по логике
- импорт проходит быстрее на локальном портале
- большие CSV обрабатываются заметно быстрее
- подготовлены конфиги и release-документы для следующего этапа
- дальнейшая публикация через Harbor и облачных админов не потребует срочного добора артефактов

## Практический приоритет

1. Ускорить текущий import execution без изменения логики.
2. Проверить локально на одном портале.
3. Подготовить release/config/documentation пакет.
4. Только потом переходить к staging/chunk architecture и cloud rollout.
